**Parallel *Software*: Basics**

We now know we have three types of hardware
1. shared memory: processor with multi cores sharing a memory
2. distributed memory system: like big super computers with cabinets and motherboards and every node has its own memory. a process running on one board cannot access the memory on another board
3. accelerators: chips and GPUs (SIMD) that are extremely good and fast at specific types of computations but bad at others.

This is why we will learn 3 programming languages/models - one for each.

## Burden on Software
From now on...
1. in shared memory systems:
	1. start a single process and fork **threads**
	2. Threads carry out *tasks*

> threads are carrying out the task. you create threads, each thread does part of the task.

2. in distributed memorys ystems:
	1. start multiple **processes**
	2. processes carry out *tasks*

> one or more processes in each boardd. these processes will also be carrying tasks (e.g. create several processes on different nodes, each one carry out parts of a task). 
> How do processes talk to each other? (they don't share memory)
> A: They talk through *messages*. This is why our first language to learn, **MPI**, is called Message Passing Interface.

3. When using accelerators (e.g. GPUs)
	1. start a process with one or more threads
	2. the thread launches task to be done on the GPU
	3. GPU will do the same task on different data

> When using accelerators like GPU, we will start with 1 or more threads (as it is a shared memory). And each thread will carry out a task. The tasks are GPU friendly. GPU will do the same task but on different data (recall GPU is SIMD chip)

these are stuff we, as software developer, has to write in parallel.

## Writing Parallel Programs
1. **Divide** the work among the processes/threads so that each process/thread gets roughly the same amount of work

> You have an algorithm or designing an algorithm to do some work. Need to divide the work among the threads or processes such that each gets roughly the same amount of work
> 
> Note: "work" doesn't just have to be computation. it could be computation AND communication.
> Note: "roughly the same" is important because *we want them to finish at roughly the same amount of time*. if they don't one fast thread/process will have to wait for the others

2. Arrange for the processes/threads to **synchronize** if needed

> sometimes you don't want to start a certain process until certain dependencies have been reached. like for matrix multiplcation, you don't want to start multiplying until both the two matrices are defined. 
> 
> Another type is, when there is a piece of code and you don't want more than one thread to be in at the same time, you use **locks**.
> 
> These synchronizations are bad for performance, but we need them for correctness. So, sometimes designing a good algorithm is to reduce the amount of sychronization neededn.

3. Arrange for **communication** among processes/threads. Reduce communication as much as possible

> Communication is expensive. So we need to reduce it as much as possible.  (remember, in order of expensive: communication -> memory access -> computation)

Here's an example of an easily parallelizable piece of software

```
double x[n], y[n]
...
for (i = 0; i < n; i++)
	x[i] += y[i];
```

they are totally independent. even though this is very easily parallelizable, but guaranteeing **load balancing** is very hard. Why? let's say we have two threads, and `x` and `y` have size 100. So we can give each thread on two different cores 50 elements each (i.e. `0-49` for thread `0`, `50-99` for thread `1`). This is perfectly same amount of computation. *However*, it will be hard to **load balance** (create load imbalance)

1. Reason: the two cores may be designed differently (multicore but hetergenous, performance core will finish much faster)
2. Other more minor rzn: one of the threads could have cache miss and other get cache hits; one have pitch falls and other won't have that; maybe for one thread one will be in cache near core the other won't be

There are things outside your control, but you can do your best (like give 70 to performance core etc.)

## Managing Threads

How many threads can you start? The whole idea depends on whether you know the problem size before starting the execution, or you know the problem size as you start the execution, or you don't know the problem size at all

**Dynamic Threads**: master threads (like the main function, starts sequentially) waits for work (e.g. program where user enters some number and multiply each number by 2), forks new threads, and when threads are done, they terminate
- Advantage: efficient use of resources
- Disadvantage: Thread creation and termination is time consuming: creating threads require operating system (every thread requires its own stack and stuff); create thread only when needed

**Static Thread**: pools of threads created and are allocated work, but do not terminate until cleanup
- Advantage: Better performance
- Disadvantage: potential waste of system resources (some therads may be staying idle doing nothing, which is waste of system resources)
*Static thread is good if you know the problem size before the program starts or from the command line*

## Nondeterminism

WHenever we are writing parallel program, we have to deal with this

**Parallelism**: you have more than one thread, each one executing a code, and you *cannot predict which program finishes first*. sometimes thread number zero will finish before thread number one. If you run that same program on that same machine after five minutes, it will be thread one that finishes first

**Race Condition**: consequence of nondeterminism. there is a race on which thread will executes first, so the result will not the same. running the program different times will get different possibilities
**Critical Section**: a piece of code where you don't want more than one thread to be in that piece of code at any point of time
1. **Mutual exclusion**: no more than one thread executes critical section at any point in time. must line up the threads so that they run critical section code one at a time
2. Need to enforce mutual exclusion through locks **(mutex, semaphore, ...)**

```
my_val = Compute_val(my_rank);
Lock(&add_my_val_lock); # blocks other threads
x += my_val;
Unlock(&add_my_val_lock); # once unlock is called, other threads that are blocked will continue
```


Q: Why, if we remove locking mechanism, we may end up getting wrong results?
A: There is nondeterminism. we dont know which thread will be at head of the queue

**Atomic operation**: all or nothing
Generally speaking, operations are not that. like `x += my_val` (there are read, write, and store). It could be that one thread needs to add 1, other thread adds 2. both needs to execute `x+=...`. So, `x` will be moving to register in first core (initially 0), and will also be 0 in the second core. The execution unit will add 1 to x, making x = 1; in the second core x will add 0 to 2, making x = 2; then first core will write x back, making x = 1; then second core will write x back, making x = 2 (overwriting what first core wrote), so the result is wrong

Atomic operation makes sure either all of it is run, or none of it
*keep in mind this worsens performance.* gotta sacrifice performance for correct result sometimes

> [!important]
> **What is the relationship between cache coherence and nondeterminism? Isn't cache coherence enough to ensure determinism?**
> Three reasons:
> 1. Cache coherence happens among caches from different cores. But if one core is hyperthreaded (it can execute more than one thread at a time), we have one cache with that core. So there is no cache coherence, and we are facing same problem as cache coherence
> 2. Critical section can be more than one statement. In that case, cache coherence may not be enough, especially since critical section may not be only for calculations. One thread may need to send something on the screen, another screen need send something different. have to make them send stuff at a different time, but this has nothing to do with cache coherence
> 3. Even if we are taking about computation (`x++`), assume `x` is shared and `x=0`. Assume two threads executing on two different cores, each one executing `x++`, each one has one cache. `x` will be read from memory to each of the caches. Then, each thread needs to add `1` to it, so `x` will move from the cache or will be copied from the cache to a register in the first core. and from the other cache to the register in the second core. *Cache coherence will NOT kick in yet, because we are still reading*. Then, inside the core, one will be added to `x` "here", 1 will be added to `x` "there". *After the addition, inside the core, cache coherence STILL WILL NOT kick in*, because we have not touched the cache yet (addition done inside the core, the registers). *Only, when one of the cores decide to wriet the results back to the cache*, cache coherence will kick in. Cache coherence will tell the first one that wanted to write `x` back to wait. Then tell the other to remove the block thta contains `x` from their cache (invalidate). Then, tell the first core it can write to the block in the cache. Now, `x=1`. Then, the other also need to write the result in the cache. But, it has removed the block, so it will get a cache miss, and so it MAY either get the inforamation from the memory or the other processor. Once it gets it, it will STILL write 1 to it, and that's wrong (it's supposed to be 2).
>    
>    Basically, even with cache coherence, because of addition and modification of `x` happening **inside the core**, cache coherence *will not* prevent nondeterminism

Simply speaking, we need locks

Summarization:
1. critical section can be more than one line in high level language, and some of them can be IO, so coherence wont even do anything here
2. two threads may be executing on same hyper-threading core, so there's only one cache. Coherence happens among caches (plural)
3. coherence will only kick in when core updates the cache, so it wont save you with computations too. 

### Busy Waiting
Locks are system calls, ∑hich means you gotta call operating system and etc. any system call will be bad for performance (gotta move to kernel mode, thread gotta be blocked, etc.)

let's try this:
Each thread has a unique ID
Assume:
- `*ok_for_1` is shared among threads
- Each thread has its own: `my_rank` and `my_val` (multi-threaded language allow you to specify which variables are shared, which are private)

assume two threads, each running the following in parallel
```
ok_for_1 = false;
my_val = Compute_val (my_rank); # Each thread will calculate its own my_val

if (my_rank == 1)
	while (!ok_for_1); # Busy waiting
x += my_val; # critical section
if (my_rank == 0)
	ok_for_1 = true; # let thread 1 update x
```
note: the first line doesn't need lock. we are not updating the variable we are just writing the same value. no matter what, `ok_for_1` will end up being false. it's only when we update it (like `x++`) that it will be faulty

At busy waiting, thread 1 will keep running while loop, while thread 0 will move down and run `x += my_val`. Then, it will set `ok_for_1` to true and let thread 1 to run

Correctness wise, *this code IS correct* for 2 threads.
Shortcomings:
1. the while loop is NOT DOING ANYTHING. when translated to assembly, the while loop is conditional jumps and what not and it will make the core busy doing nothingn and that's wasting resources
2. not very easy to scale the code beyond two threads.

# How to write a Parallel Program?
We have a serial program, how do we parallelize it?
Goal: divide work, ensure load balancing, manage synchronization, and reduce communication
There is *no mechanical process*. in the end its gon be trial and error etc.
**First**, profile code to find **hotspots**
- parts of code that are taking most of the execution time, such as a function that is taking so much time then we need to focus our attention to that)

> Note: write from sequential first always, then write parallel program. Otherwise, how do you know the parallel code is good? Need a benchmark for comparison.
> Also, parallelizing a sequential code is easier than writing parallel code from scratch

**Second**, parallelize the hotspots
- note: its not just about the big O anymore, cuz like algorithms like quick sort with good big O is veyr hard to parallelize, while bubble sort which has terrible O is easily parallelizable.
- Ian Foster has some nice framework, described in his book "Designing and Building Parallel Programs" (the specifics outdated or something but framework is good)

## Foster's Methodology (PCAM)

**1. Partitioning**: divide the *computation* to be performed and the *data* operated on by the computation into small tasks. Simply speaking, look at hotspot and try to split it up into as many smaller tasks as possible.
- 
This step brings out the parallelism in the algorithm

A checklist for problem partitioning:
1. does partition define at least an order of magnitude more tasks than there are processors/cores in your target computer? If not, your design may not be scalable and/or lose performance

> two tasks in parallel may not cancel out the cost it took to create the threads or processes and make them execute in parallel. need way more tasks than number of cores

2. Does your partition avoid redundant computation and storage requirements? If not, the resulting algorithm may not be scalable to deal with large numbers

3. Are tasks of comparable size? If not, you may face load balancing issues later.

4. Does the number of tasks scale witht he problem size? Rule of thumb: *an increase in problem size should increase the number of tasks rather than the size of individual tasks*.

> When you have a bigger problem, you are increasing the number of work per thread, your code is not the best. a good quality parallel code is, when the problem gets bigger, you can get more threads or more cores.
>
> Example: we have a list and we want to determine number of ways each number appear
> Strategy 1: one thread responsible for counting number of for one character. but, what happens when the string gets longer and longer? well, each thread needs to work more.
> Strategy 2: take that string, take it into pieces (4), and give every thread a piece. every thread will be counting its numbers. If we have bigger strings, we can just cut it up more and each thread won't rlly be doing more work.

5. Have you identified several alternative paritions?

> e.g. we know floating points take more time than ints. Multiplications and divisions take longer than additions and subtractions.
