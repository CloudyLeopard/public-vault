## Foster's Methodology (PCAM)

**1. Partitioning**: (see previous class' notes)

**2. Communication**: determine what communication needs to be carried out among the tasks identified in the previous steps

The second task depends on the previous tasks, and so will require communication.

When you start drawing arrow and realize most of the task depends on a certain task, we can't really parallelize it. We may have to consider another algorithm (the best algo for sequential program may not be the best for parallelization)

Checklist for communication
- Do all tasks perform about the same number of communication operations? (each one is doing roughly the same number of communications, like same amount of arrows going out of it.) Unbalanced communication requirements suggest a nonscalable construct (if we get a bigger machine, we still will not get a bigger performance; or, if the program gets bigger and bigger, performance go down by a lot)
- Does each task communicate only with a small number of neighbors? (if a task is communication with several other tasks and are not next to each other, once we put them in a real chip, if your core is communicating with the next neighbor its fine, but if its with a core thats far it will take much longer)
	- If each task must communicate with many other tasks, evaluate the possibility of formulating this global communication in terms of a local communication structure. (if we have to broadcast something, maybe reformulate the algorithm. Or, could we recalculate the data instead of getting data from another core - recalculation is much cheaper than communication)
- Are commuication operations able to proceed concurrently? If not, your algorithm is likely to be inefficient and nonscalable.

Once everything is clear, translating to code is trivial

**3. Agglomeration or Aggregation**: Combine tasks and communication identified in the first step into larger tasks

this is the step where you tihnk can you combine two tasks or not. Aggregation is putting tasks together for the sake of reducing the amount of communication.

For example, if task A must be executed before task B can be executed, it may make sense to aggregate them into a single composite task. 

Checklist for agglomeration
- Has agglomeration reduced communication cotsts by increasing locality? If not, exaine your algorithm to determine wehther this could be acheived using an agglomeration strategy
- Have you explored replicated data or computation to reduce communication cost and explored the benefits and costs?
> At least a few extra computation to reduce communication will pay off. 
- Has agglomeration affected load balancing in a negative way? You may need to check several agglomeration alternatives
> Some tasks much bigger than another will have load imbalance. And, we mean computation AND communication

**4. Mapping**: assign the composite tasks identified in the previous step to processes/thread

Time to make the first decision: should we surround these things using processes and communication with each other (MPI), or have one process and multiple thread and communicate use meomry (OpenMP), or use an accelerator (CODA)

This should be done such that
- Communication is minimized
- Each process/thread gets roughly the same amount of work (load balancing) (note, same amount of *work*, NOT *computation*)

| Threads                                                                                                                                                                                             | Processes                                                                                                                                                                                                                                                                                        |
| --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| Smaller Sized Tasks <br>1. much cheaper to create threads than processes.<br>2. the heavier the communication among the task, then its better to use threads. communication among processes is slow | Bigger tasks                                                                                                                                                                                                                                                                                     |
| Larger amount of communication                                                                                                                                                                      | Smaller amount of communication                                                                                                                                                                                                                                                                  |

Number of tasks is *not enough of a factor* to tell us processes vs threads. It is the size, communication, AND type of tasks. 

Assigning tasks to processes threads can be done using:
- Dynamic threading
- Static threads
- Single Program multiple Data (SPMD)

Need to decide on whether to use
- one process with several threads
- Several processes with one thread each
- Several processes with several thread each

## Concurrency vs Parallelism
They are *not* the same.

**Concurrency**: At least two tasks are making progress at the same **time frame**.
- Not necessarily at the same time. Maybe during that 1 minute, only one core will work with one task for a while, and work with another task after that.
- Include techniques like time-slicing
- Can be implemented on a single processing unit
- Concept more general than parallelism (it includes having only one core)

**Parallelism**: At least two tasks *literally* execute at the same time
- Required hardware with multiple processing units

====Parallelism is a subset of concurrency.==

Code is concurrent if its pieces are independent (doesn't matter if task 1 goes first or task 2 goes first)

## Amdahl's Law
Rule of thumb - **Amdahl's Law**: How much of a speedup one could get for a given parallelized task?
If $F$ is the fraction of a calculation that is sequential, then the maximum speed-up can be achieved by using $P$ processes is 
$$\frac{1}{F + \frac{1-F}{P}}$$
$P$ is only parallelizing the parallelizable part. There is a part that is only sequential and cannot be parallelized.

![[amdahl's law visual rep.png]]
*You are only parallelizing the parallelizable part*

The theoretical upper bound is that you can never reach is, when $P\rightarrow \infty$, $\frac{1}{F}$

#### Derivation fo Amdahl's Law
$Tseq$ = time taken to execute the program on a sequential machine
- depends on architecture of hardware, the problem size, and algorithm
$Tpar$ = time taken to execute the program on a machine with $p$ cores
- Depends on parallel architecture of hardware, problem size, and parallel algorithm
$F$ = fraction of the program that is purely sequential
$$Tpar = F\times Tseq + (1-F)\frac{Tseq}{p}$$
$$Speedup = \frac{Tseq}{Tpar} = \frac{1}{F + \frac{1-F}{P}}$$

#### Main point of Amdahl's Law
1. Don't invest blindly on large number of processors (before you do your best to reduce $F$ first)
2. Sometimes, having faster core (thru pipelining, superscalar, hypterthreading, etc.) or processor makes more sense than having many
	- This is because having faster core will speed up and reduce sequential part (the pink part in the above diagram). Finish sequential part quickly.

Was he right?
- In his days, many programs have long sequential parts. But, **this is not necessarily the case nowadays** - more stuff can be parallelized now.
- It is not very easy to find $F$ (sequential portions)

> Amdahl did not give any rule on how to calculate $F$. If we are given an algorithm (pseudocode), how would we calculate $F$? There is no one correct answer. There are several things:
 - Back of the envelope algorithms: you have several algorithms and you don't want to translate them into the code to calculate them.

Additionally, overheads that Amdahl's Law ignored
![[amdahl's law overhead.png]]
All the communication stuff is *not taken into account*

- Impact of sequential-to-parallel synchronization and inter-core communication: also, more threads = more work (cuz communication). So, the more parallelism we have, the more our performance may suffer when it comes to sequential to parallel and enter core communication.
- In applications with high degree of data sharing (all these threads need to commuicate data, e.g. with a global variable), will lead to intense intercore connectivity requirements. That work load should be executed on a smaller number of **larger cores**. Instead of having many weaker cores, better to have fewer, larger cores (cores with pipeliing, superscalar capabilities, hyperthreading technology)
	- larger cores will decrease communication which is great. A core that is four-way hyperthreading (four different threads in one core), then communication will be greatlyr educed - all of it will be done inside that core.
	- Communication si way more expensive than hyperthreadig threads competing for resources
- For tasks with high-intercore commuication requires, the workload should be assigned to small number of cores, even if the parallelization fraction is close to 1
	- the more communication, the better it is to have less bigger cores.
- For low arithmetic intensity tasks (less computation, more memory access or IO), the workload may better be assigned to one core, even if parallelization fraction is close to 1
	- since they have low arithmetic intensity, they won't be using much of the execution unit. Assigning it all to one core will be better.

Bottom line: not just about parallelization, but about reducing data communication. 
- Try not to use lots of global variables


### Take Away
Decreasing the serialized portion is of greater importance than adding more cores blindly

Only when program is mostly parallelized (smallest $F$ we can get), does adding more processes help more parallelizing the remaining rest

Amdahl does not take into account of
- Overhead of synchronization, communication, OS, etc.
- Load may not be balanced among cores
*Only use this law as guideline and theoretical bound only, not for just speed up.*

## Determining Efficiency
**Can we analyze how good a parallel program/algorithm is without executing it on a machine?**

DAG Model for multithreading
![[DAG model for multithreading.png]]

If algorithm has a loop, and every iteration is a different instruction. The graph has to be **acyclic** and **finite**.

A vertex is a unit of execution. 
For example: 
- An instruction (from now on we assume all vertices are an instruction)
- A basic block
- A function
- Any granularity you decide on 

An edge indicates dependency. For example, an edge from vertex A to vertex B means A must execute first, then followed by B.

For now, we assume that every vertex/instruction takes the same amount of tim eunit

**Work**: total amount of time spent on all instructions

> In the DAG above, "work" is **18**. It's basically total amount of time running it sequentially.

$T_P$ = fastest possible execution time on $P$ processors
**Work law**: $T_P \geq T_1/P$
- $T_1$ is work (purely sequential execution) - basically time using 1 processor/core
In 99% of the time, this work law will hold. This is because $T_1/P$ assumes linear decreasing in time that has not been taking into account - things are not as linear as youd think. So $T_P$ is usually bigger, but in rare cases it is the same.

**Span**: longest path of dependence in the DAG. *Here, we assume every vertex takes the same amount of time.*. it is a **timing measurement**: the longest path that depends on timing
- here, it is 9 (from 1 to 18) time unit (cuz each vertex is 1 time unit) 
- Suppose like 15 takes 20 time unit, then the span is 25
- = $T_\infty$ (with infinite processors will still take 9 time units)
But note that THIS IS DEPENDENT ON TIME UNIT - if a instruction takes longer time unit itll be different. not just about longest path

**Span Law**: $T_P \geq T_\infty$
$T_1 \geq T_2 \geq T_3 \geq \cdots \geq T_\infty$
