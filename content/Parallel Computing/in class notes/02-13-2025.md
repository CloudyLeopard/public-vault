## DAG Model for Multithreading
this is a way to evaluate a piece of code without actually running it. If you have several algorithms and you want to see which one to move forward with, a good idea is to draw the DAG, analyze it, and determine which one to use. (Better in terms of which one has better parallelism, etc.)

Freedom in determining what a vertex represents - a function, a statement in high level language, etc. But you have to be **consistent**

For now, we assume every vertex is an instruction, and every vertex takes 1 unit of time. In real life, different vertices will take different amount of time.

Note: the graph is *acyclic*. If the algorithm has a **loop**, you will have to like "run" it a couple of iterations (2-3) and unfold it to analyze the DAG.

Arrows represent dependencies - a later vertex cannot execute until its parent has finished.

Still, even with **span** and **work**, it's not enough to analyze the DAG.

Introducing **Ratio of work to span**
$$T_1 / T_\infty$$
The higher the ratio, the better! The speedup is the T sequential of the program divided by the T parallel (we will learn this later). Simply speaking, $T_1 / T_\infty$ is the maximum possible speed up that can be created by any number of cores/processors. Higher = more speed up potential (i think)

Example:

6.25 is the highest possible speed up *given by any number of processors*. The rzn its not "8" (cuz span is 8 or something idk) is because the starting and ending node takes time.

Let's assume the **parallelism** is 6.25. What is the minimum number of cores that can give me the best speedup?
Sequential is 50. With 8 cores, you get 50/8=6.25. So, you will never get any speed up higher than this. If we use 7 cores, we will get a little bit lower: 6 units of time using 7 cores, and then 6 more units of time again after the 7 cores is done (i think)

Note: using 7 cores will give you the exact same speed up as using 4 cores. But $T_8$ will give you the highest one.

### Amdahl's Law vs DAG
DAG's more precise! But Amdahl's law is easier to calculate. Use Amdahl's law first to make first elimination, then use DAG to find out what is the better method for the remaining algorithms.

#### Coherence
The three things that made coherence bad
1. Extra bandwidth (scarce resource)

2. Latency due to the protocol.

> The directory is telling other memories to invalidate blocks and stuff. The core that is updating its own cache has to wait until all the invalidating is done. Those who has invalidated the cache block may have cache misses, and that takes more time later.

3. False Sharing

Even though when you are updating different variables, the cache coherence thinks you are updating the same thing *because machine only recognizes cache blocks not variables*

> Suppose we have an array initialized with some integers. Now, we want to increment every element in the array by 1. In sequential code is easy. But, how do we parallelize with two threads?
> Strategy #1: Give elements 0-15 to thread 0, and 16-31 to thread 1
> Strategy #2: Give even elements to thread 0, and odd elements to thread 1
> Both strategies have the *same degree of independence*, and both *have the same amount of calculations*. 
> However, **strategy #1 will be *way* faster**

> Strategy 2: thread 0 will need to get `A[0]`, get cache miss, and go get the **cache block** that contains `A[0]` from the memory (usually a cache block has 64 bytes, so 16 ints). Now, `A[0...15]` will be in the cache. But now, thread 1 will need to get `A[1]`, get a cache miss, and get the entire cache block from `A[0...15]` to thread 1 too.
> Now, the cache coherence hardware *does not understand variables*. It **only understands cache blocks**. Cache coherence will kick in when both threads try to update `A[0]` and `A[1]`.

> Strategy 1: in 201, we learned "alignment". So, the first 16 elements will be in one cache block, and the second 16 elements will be in another. So, cache coherence will not kick in.

> [!warning] TF is alignment

Note: this type of improving efficiency will not be done by the compiler. Hence why this is important to know

### Load Balancing
![[Load balancing.png]]

The more synchronization points you have, the more sensitive you are to load imbalance.
Gotta do two things:
1. Try to reduce synchronization points
2. Try to reduce load imbalance (not just computation, but number/pattern of memory access, and communication)

When there is load balance, they will finish roughly finish in the same time. 
So, you want to assign to the "right" core: like give the task that has more computation to the performance core, and the task with less to the efficiency core. *therefore, even if it looks like there is load imbalance, there really isn't. **What's important is the time they finish***

If you cannot eliminate it, at least reduce it.

## General Advice
One advice in creating parallel code is **first start by writing purely sequential code**, then **translate to parallel code**

Then start testing as follow:
1. Test parallel code with only one thread. When you test with only one thread and debug it and make sure it works correctly, *you debug all the sequential bugs*. (note, the one thread version should usually be slower than sequential version, cuz overhead).
2. once 1-thread works correct, test with *two threads* only.
3. If that works correctly, test with *four threads*

### Divide-And-Conquer
![[divide and conquer diagram pc.png]]
split split split then merge merge merge

The high level of parallelism will happen *in the middle of the picture*.

**When do you stop splitting?**
Depends on number of cores, and also how much work per thread - you want to give every thread a decent amount of work. This takes some trial and error.
After you finish and solve the sub-problems in paralle, you start merging.

There is some **overhead** here - you are using extra time during split and the merge. Because of this (which are all sequential things), you *will not get 4x speed up* (4 in reference to diagram)

**If the problem size is not big enough, the sequential solution will always be faster than the parallel version**

Some problems are good for divide and conquer.
