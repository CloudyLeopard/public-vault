## Pipeline
A series of **ordered** computation stages need to be applied on data
*do not get confused with core's pipelining!!!*

Suppose you have a for loop. The loop iterations are totally independent from each other, it is easy to pipeline. This is called **loop level parallelism**, and we don't need to talk about this cuz its very straightforward.

But what if they are dependent on each other? Then you cannot parallelize. You may either get incorrect errors, or you have to create barriers and that just means making more api calls and making things slower.

When loops are not independent of each other, there is a way to extract a sub-parallelism from it, and that is called **pipelining**

![[pipeline parallelism.png]]

In this picture, every line is *one loop iteration*. We look at every loop iteration, and we cut it into 6 pieces here. They are also ordered (C1 then C2 then ... then C6). 

Because we have 6 pieces, we will need 6 threads. One thread will be responsible for C1, one will be for C2, etc. And when a thread starts doing C2, the first thread will start doing C1 in the second iteration. After that time stamp, the first thread will start doing C1 in third iteration, second thread will start doing C2 in second iteratino, third thread still start doing C3 in first iteration. **Every thread will be responsible for some parts**

This is not as good as loop level parallelism obviously, but its better than purely sequential.

Every line is one loop iteration!

Useful for 
- streaming: the file that contain the song or the movie will go through several stages before it is shown to you, very much ammenable for this type of pipelining
- Loops that are hard to parallelize
	- due to inter-loop dependence: there are some dependencies between iterations. Instead of totally giving up, we will try this pipeline
> What if the first thread cannot start C1 because its waiting on something from C6? Then, you cannot parallelize at all. You will have to find a different algorithm.

How to do it?
1. split each loop iteration into independent stages (e.g. C1, C2, ...)
2. Assign each stage to a thread (e.g. T1 does C1, T2 does C2)
3. When a thread is done with each stage, it can start with the same stage for te following loop iteration (T1 finishes C1 of iteration 0, then start C1 of iteration 1, etc.)

Advantages
- Exposes intra-loop parallelism
> if you have both inter loop and intra loop, you pick inter loop (cannot have both)
- Locality increases for variables used across stages
> In the example above, thread for C1 is really the same code. so it will use the same variable, and get a better performance (cache hit, locality)

### Example of Pipeline Parallelism
```
// compression algorithm
while (!done) {
	Read block;
	Compress the block;
	Write block;
}
```
While not done, read a block of few bytes of data, compress the block, then write the block. Then get next block, read, compress, put it back, etc.

> Q: Is **inter** loop parallelism possible?
> A: No. You need to compress the file and put it in the file *after* the previous part. The compress block has to be put in the destination file in order. For correctness reason, we cannot execute the iterations in parallel

If we look at the iteration, there are three distinct instructions. And it has to follow the same order. Thus, we have potential for pipeline

### How to Avoid Race Conditions:
**Critical section**: a part of the code that is modifying some shared data (*shared memory*), and you want at most one thread to be there at a time. If two threads are there at the same time your code may be wrong

Prohibit more than one thread from reading and writing the shared data at the same time --> **mutual exclusion**

> Q: Why do they have stack and heap go towards each other?
> A: Once the program is starting, we know exactly the size of the text, data+bss, but we do not know the size of the heap and the size of the stack (we do not know what dynamic allocation the program will do, we do not know how many nested function calls we will do). in order to not lose any memory, the best way to put two unknown entities is to put them grow toward each other. (yea this makes sense)

We have two threads because in multi-threaded programs, every thread has its own stack. Every thread needs its own stack because
in multi-threaded programs, the threaded programs will not always execute the same code. every thread may be executing different code, which means it may be calling different functions and so on. ## Sources of Parallel Overheads
we want to reduce the overhead

- overhead of creating threads/processes
> these are system calls, which has overhead. because of this, for multi-threaded code, it costs much less to call the operating system once and tell it to create 10 threads for me, then call the operating system 10 times and each time to ask for one thread. (note, these are under human control)
- Synchronization (barriers)
> each time you have a barrier, all the threads/processes must reach it before they are allowed to proceed. You are slwoing down threads/processes, thus lose performance
- Load imbalance
> at the end of the program, we will not finish until everybody finish. if you have load imbalance, one of them is much slower, so you have to wait for it.
> if that thread has way more work than the otheres, that means CPU will be doing a lot of work/calculation, and it becomes hot, then dynamic voltage and frequency scaling will kick in, slowing down the core to make it avoid burning. 
> So not only is everybody waiting on the slower thread, it will make the CPU slower
- Communication
> again, communication is the slowest thing. For shared memory, communication also exists, but it exists implicitly (going to the cache and come back, coherence and invalidating stuff, level 3 cache miss and going to motherboard and com eback, etc.)
- Extra computation
> we said before that computation is a distant third when it comes to performance (communication -> memory access -> computation)
> However, this does not mean that parallel program will ahve 10 times the cmoputation fo sequential program because you want to expose more parallelism. it will still take time
> We don't mind some extra computation, but not that much. Sometimes, instead of one thread send data to another, we will make another thread do some more calculation cuz sending communication si slow. but if we do this on a large scale, we are makinag tons of redundant extra calculation adn that may slow things down
- Memory access
> You need to access memory sequentially. cache friendly code. also write code that reduces a little bit due to coherence kickingin from false sharing and so on.

### Efficiency
$$E = \frac{Speedup}{p} = \frac{T_{serial} / T_{parallel}}{p}$$
$P$ can be number of threads, processes, or cores
> he will tell us what P is in the exam or program

The higher efficiency the better! This equation says, speed up using p cores. As you keep increasing number of cores, the speedup should increase, and p goes up (cuz p is number of core). If E in general increases, that means the speedup grows faster than p

efficiency is a measurement of how well you are doing with the resources you have. If speedup is growing faster than p (e.g. 2 core than 1, and speedup grows faster than 2). Efficiency is how well you are making from the resources you have.

### Be Careful about $T$
Both $T_{serial}$ and $T_{parallel}$ are **wall-clock times** (look at watch, then start program, and when it finishes look at clock again and see how long it takes). These are not fully under your control as a programmer. 
- they are influenced by you (this will have less time than when you have a better parallel code). 
- Influenced by choice of compiler
- depends on the compiler switches (eg turning optimization on/off)
- The operating system: if operating system is very efficient at creating threads/processes, you're gonna get a better $T$
- the type of filesystem holding the input data
- The time of the day (different workloads, network traffic, etc.)
