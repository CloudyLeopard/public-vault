## Scalability
How does your parallel program behave if I give you bigger problem size? or if I give you more core or more processes or more threads?

**Scalability** is the ability of a (software or hardware) system to handle a growing amount of work efficiently. For this course, we concentrate on software.

> Bigger problem size = t_serial will increase, and t_parallel may increase a little bit

If we keep the efficiency fixed by increasing the number of processes/threads and without increasing problem size, the problem is *strongly scalable*.

> Suppose we have a parallel problem, and I keep the problem (e.g. matrix mult), and we increase the number of processes/threads but keep the problem size fixed (same matrix size). if i give you more cores/processes with the same problem size and the efficiency is fixed, it means that if I give you more core, you can give them a lot more work to do. Simply speaking, the t_parallel will decrease such that even when I increase p, the speedup will increase so the efficiency will become fixed.
> We increase P. If E remains the same, then speedup must also increase at the same rate as we increase P. How can speedup go up at the same rate? *T_parallel must decrease* (T_serial doesn't change because we maintain the same problem size). This means the problem is *strongly scalable*: more p leads to a much lower parallel time at the same rate.

If we increase P, but in order to keep E fixed, we also have to increase the problem size at the same rate as we increase the number of processes/threads, the problem is *weakly scalable*.

> increase problem size --> increase T_serial

>Q: How to improve efficiecy?
> A: Sometimes you cannot improve efficiency. You are getting parallelism but every thread/core is doing a little work. Efficiency does not measure how fast your code is, but how well your processes are using resources.

### Execution Time
**Elapsed Time** (aka **Wall-clock time**):
- Includes *everything* (execution time, I/O waiting, context switches by OS, disk and memory access, etc.)
- A useful number, but often not good for comparison purposes
> Not good because there are many things outside of your control as a programmer. You may have a wonderful code but your elapsed time because it depends on what other processes are running on the machine. 
> Example: if we wait to submit the hw at the last minute maybe everyone is submitting at the same time and context switching slows down the program

**CPU Time**
- Doesn't count I/O time or time spent running other programs
> measure how well the cores have taken to execute your code. The counter stops counting if code is removed from CPU due to I/O or context switch.
- Can be broken up into system time, and user time
	- System time: time taken by CPU to execute system call to serve your code
	- User time: time taken by CPU to execute your code
> It includes not just your code, but also OS time that serves your code. For example, if you create a thread (system call), the operating system will start executing something and this is counted in CPU time.

Our focus: **user CPU time**
- time spent executing the lines of code that are "in" our program

`clock_t clock(void)` returns the number of clock ticks elapsed since the program started.

```C
#include <time.h> // <-- include this to measure time
#include<stdio.h>

int main() {
	clock_t start, end, total; // <--
	int i;

	start = clock(); // <--

	for (i = 0; i < 10000000; i++) {}

	end = clock(); // <--
	total = (double) (end-start) / CLOCKS_PER_SEC; // cycles/cycles_per_sec

	printf("Total time taken by CPU: %f\n", total)
}
```

Note: `(end-start)` is the total number of cycles. Divide by `CLOCKS_PER_SEC` to get time

### Execution Time for Sequential program

## Pitfalls of Timing in Parallel Programs

### For Multithreaded Programs
1. The total number of instructions executed may be different across different runs
> The effect increases with number of cores. Suppose we have the same program, running on same machine with the same input. We run that program several times. The discrepancy will get bigger as number of core increases
> A: increasing number of cores = need to generate more thread = ask OS to generate more thread = ask OS to do more work = total number of instruction may increase. 
> Hence, you may need to run your program several times to get to what is average as close as possible

2. System-level code account for significant fraction of the total execution time

Also, your **program does not run in a vaccume**
- OS at least is there
- Multi-programming and/or mulithreading setting is very common in multicore settings
> Programs can be multithreaded. We can have several programs running in the background running *with* your code.
> These programs will obviously affect your code's performance, not just because of competing for cores, but also...
- Independent programs affect each other
> Cache stuff, competing for memory, etc. This is another reason why you gotta run program several times and get average
