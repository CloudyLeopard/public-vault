## MPI Processes

Every process inside MPI has a unique ID called a rank

- Identify processes by non-negative integer **ranks**
- Each process has a unique rank
- `p` processes are numbered 0, 1, 2, ...., p-1

If all the processes we create are doing the same thing, it is redundant. But here, notice that all the processes have the same code. So, you need to put if-else and if rank=0 do this, if rank=1 do that. This is why we have rank, so we can pinpoint different processes and make them do different things.

### Communicators:

In its simplest form, a communicator is a **collection of processes** that can send (and receive) messages to each other.

> you, as a programmer, can create these communicators. Suppose you create 10 processes, and you want the first 5 processes to be a group, and the last 5 to be another. You create 2 communicators (we will see how later)

`MPI_Init` defines a (one big) communicator that consists of all the processes created when the program started

> It will have a communicator that will contain all 10 processes. Every communciator must have a name. 

This communicator is called `MPI_COMM_WORLD`
> this is why earlier in the example code we have the name `MPI_COMM_WORLD`, even when sending and receiving. It is a very important name of the communicator. It's almost always an argument in any MPI api calls.

Given this, we have two important APIs related to the communicator

```C
int MPI_Comm_size(
	MPI_Comm comm // input
	int* comm_sz_p // output generated will be put here
)
```

it will return the number of processes in that communicator into `comm_sz_p`

```C
int MPI_Comm_rank(
	MPI_Comm comm // in
	int* my_rank_p // out
)
```

`my_rank_p`: will be the rank of the process making this call

> [!important]
> All the MPI APIs return an `int`, which tells you if it succeeded (positive number) or failed (negative number).

> Q: You execute the program with `-n 10`, so we know we have 10 processes. Then why do we need to have another API that will tell us the number of processes?
> A: its just as the programmer, you may not know how many processes there will be (the programmer is not the user)

### Communication

```C
int MPI_Send(
	void* msg_buf_p // int, pointer to beginning of memory im sending
	int msg_size // in, num of elements in msg_buf
	MPI_Datatype msg_type // in, type of each element in msg_buf
	int dest // in, rank of the receiving process
	int tag // in, to distinguish the messages
	MPI_Comm communicator // in
)
```

**Whenever there is an `MPI_Send`, you MUST have an `MPI_Receive`**

Message sent by a process using one communicator cannot be received by a process in another communicator

NOTES:
- `msg_buf_p`: the pointer tells the MPI run time where to start looking for the message to be sent to the destination
- `msg_size`: is **NOTE** the size, but the NUMBER OF ELEMENTS we want to send
- `msg_type`: tells the MPI run time how big each element is really (float, int, etc.)
- `tag`: suppose I am sending two messages from process 1 to process 2. If I send the first message then the second message, *there is a chance they may be receive out of ordered.*. The receiving end knows which is which by using the tag, which is a positive number (e.g. 0, 1)
- `communicator`: the name of the communicator. Only processes in the same communicator can talk to each other. (right now, we only know of the big one `MPI_COMM_WORLD)

`MPI_Datatype` is strange - it doesn't specify any datatype. Recall you cannot have the "type" as an argument on its own.
--> In order to go around this, `MPI` gives a name of type that corresponds to the `C` type.

One thing to keep in mind, `MPI_Recv` (in all the MPI implementation) is *blocking*. So, whenever a process runs `MPI_Recv`, it will block!. So if you have an `MPI_Recv` without an `MPI_Send`, you will be blocked forever (deadlock)

Different companies have different MPI implementations, but all follow some industry standard. The standard says `MPI_Recv` must be blocking, but doesn't say anything about `MPI_Send`, so there are different implementations.
