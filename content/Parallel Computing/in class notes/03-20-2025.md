## Safety in MPI Programs
A program that relies on MPI provided buffering is said to be **unsafe**

> MPI Send in blocking for some implementations, unblocking in others. In fact, sometimes MPI Send is blocking up to a certain size, until its not blocking (or the other way around idk)

> If two processes both call MPI Send, If MPI is nonblocking, there's no problem. On other hand, if MPI is blocking, then both processes executing MPI send will be stuck on other partner waiting for MPI Recv, but the other partner is waiting for the previous partner to call MPI Recv -> deadlock

> A program that relies on MPI provided buffering means a program that relies on MPI send to be non-blocking is said to be **unsafe**: we will not guarantee it will work (i.e. will not reach deadlock). It is up to implementation, not part of the MPI standard.


Such a program may run without problems for various sets of input, but it may hang or crash with other sets.

> This is **unsafe** because such a program may run without problem for various sets of inputs (e.g. small inputs), but may hang or crash on other cases.

> one way to do this is one process do send and receive, the other do receive and send. But first, that is for send/receive.

Let's try to generalize first: if you are using PMI send a lot in your program, and you want to make sure the program will not crash or deadlock for any input, we can do this:

## `MPI_Ssend`

The extra `S` means synchronous, and synchronous means blocking. the arguments of the MPI Ssend is exactly same as MPISend, but MPI Ssend is always blocking

MPI_Ssend is guaranteed to block until the matching receive starts.

How can this guarantee program is safe?
- Replace all MPI_Send calls in your code with MPI_SSend
- If your program does not hang or crash -> the original program is safe, and you can go back to MPI_Send.

## `MPI_Sendrecv`

You do not decide who sends first then receive, you delegate this to the MPI Runtime

An alternative to scheduling the communications ourselves

Carries out a **blocking send and a receive in a single call**

Especially useful because MPI schedules the communications so that the progarm won't hang or crash
> Because MPI schedules the communication, MPI runtime knows what type of interconnect is there among different nodes, which one has contention, etc. it will guarantee performance and guarantee no hanging or crash. it is by definition a non-blocking call

use this when: replace all pairs of consecutive send and receive calls. 
> Use this when there is mutual send and receive - when two processes needs to exchange data among themselves.

## The Communicators

Last new things with MPI. Until now, we are using the default communicator called `MPI_COMM_World` or something. it contains all the processes in the program. Sometimes, we have to create communicators that are subsets of this.

applications on where having a smaller communicator is needed:

since we can create different communicators, then a single process can be a member of more than one commuciator, and that process will have a unique id/rank in each one of these communciators. and these ranks can be different! the same process can have a rank of 7 in `MPI_Comm_world` but a rank of 1 in a new communicator 1, and so on. 
So we have to be very careful about this. a single process can have different ranks about different communicators.


A **communicator** can be thought of as a handle to a group of an ordered set of processes

> Any processes will start from rank 0 to number of processes in that communicator.

For many applications maintaining different groups is appropriate
> maybe one group deal with ai other group deal with sound other group deal with visual etc.
> or in science, one deal with stars, another deal with movement, another deal with energy, etc.

Groups allow collective operations to work on a subset of processes.
> e.g. `MPI_Scatter`, ...

### `MPI_Comm_split`

Before you create, you have to give the communciator a name. You can give it any name -follows the rules of variabl enames

```C
int MPI_Comm_split(
	MPI_Comm comm, // called by all processes in comm
	int color, // must be non-negative
	int key, // rank of the process in newcomm
	MPI_Comm * newcomm
)
```

Partitions the group assocaited with comm into **disjoint subgroups**

Processe with the **same color** will be in the same group
> color means all the processes who called mpi_comm_split with the same color will be in the same communicator.
> - if two processe call with color 2, three processes call with color 3, we have 2 communicators (excluding the big one)

Within each subgroup, the processes are **ranked in the order defined by the value of the "key**
- with ties broken according to their rank in the old group

> what if a process puts key (rank) 7,a nd the other puts 8? well, the one who puts the smaller rank will be rank 0, and the bigger one will be rank 1.
> If the two processes call the same rank, mpi will choose based on the rank in the original communicator.

**the original communicator does not go away!**

> **Example**
> Q: Suppose MPI_Comm_world has 4 processes. And then, the 4 processes all call on `MPI_Comm_split`. two of them use `color=1`, the other two use `color=7`.
> After calling MPI_comm_split, how many communicator will we end up having in the whole system?
> A: 3. we have `MPI_Comm_world` + two communicators
> 
> Follow up Q: How many names do we have for communicators?
> A: `MPI_Comm_split` will create number of communicators = number of distinct color. All of the communciators will **have the same name**. The MPI under the hood will be able to differentiate, but they all have the same name defined by `newcomm`

If a process uses the color `MPI_UNDEFINED` it won't be included in the new communicator

You could just leave it there and wait until program ends, but mpi runtime keeps tracks of all the program so removing it from mpi run time saves time.

## Words of Wisdom!

### When to use MPI

Portability and performance
> portability across different machines (e.g. all the top 500 super computers can run MPI). Almost a defacto when it comes to processe that are collaborating

Building tools for others
- libraries

Need to manage memory on a per process basis
>  for example, suppose you have a process where you want a large heap but a small stack, so we make huge dynamic allocation but not a lot of nested calls; in other cases its the opposite, i have a large stack but id ont need a large heap, etc.

### When not to use MPI

Programs that have irregular communication patterns are often difficult to express in MPI's message passing model
> Irregular commnuication pattern means you will not be able to use collective calls. When you have something that doesn't have specific pattern, you can only use point-to-point, which means you will be calling mpi_send and mpi_recv a lot, which will make performance go down

Domain-specific applications with an API tailored to that application
> e.g. streamlit is for streaming applications. usually when you have a programming language is made for a specific type of application, you can't have mpi beat a tailored language to that application

Require Fault Tolerance
> ther eis some research on how to make MPI fault tolerance, but based on what we have seen, it's not. What if in a big super cmoptuer it crashed? As soon as you have a mpi collective call, we will have a deadlock.
> MPI is not created in such a way to have a fault tolerance.
> Fault is a very big field in computing. E.g. 1 bit is flipped from 0 to 1 due to dusts, cosmic rays from the sun, etc. this is especially needed now, now that transistors are really small

### Strengths of MPI

small
- many programs can be written with only 6 basic functions

But also large
- MPI's extensive funtionality (MPI-1 contains about 125 API, let alone MPI-2 and MPI-3)

Scalable
- point-to-point communication

Flexible
- don't need to rewrite parallel programs across platforms

### MPI Features we did not cover

Dynamic process creation
> here, processes are created once with `mpi_exec` and with `mpi_run`.

One sided communications

Groups and their relations to communicators
> they are not the same, even tho related.

# OpenMP

MPI is defactor for distributed memory system, openmp became defactor for multi-threaded programming. 

15 years ago, instead of openmp we would have discussed pthread. but openmp is much more userfriendly, and it now also works with accelerators (we can write code to gpus). But we will be using it here for traditional shared memory machine.

In order to see this trend and importance of openmp, we need to see a program that is highly computationally intensive:
```C
#include <stdio.h>
#include <stdlib.h>

int main() {
	// do this in parallel
	print ("Hello, World!\n");
	
	return 0
}
```
*lmfao this prof istg hes so good*

we add this
```C
#include <stdio.h>
#include <stdlib.h>
#include <omp.h> // <--

int main() {
	omp_set_num_threads(16); // <--

	// do this in parallel
	#pragma omp parallel // <--
	{
		print ("Hello, World!\n");
	}
	
	return 0
}
```

`omp_set_num_threads(16)`: we want 15 more threads (cuz we already have 1 thread that exists). but, no threads are created yet here

Most of what we will learn about openmp will start with `#pragma omp ____`
- `parallel` is the key word: it means **create threads**.
- we have already told the runtime to have 16 threads, so this line will make 15 more threads.

Each of the thread will run the following structure piece of code
- here, each 16 thraeds will run the print statement
- but outside of the structure `{}` the stuff will be only run once

**basic idea of openmp**
![[openmp initial idea.png]]

if we have 1000 loop iterations, and we have 10 threads, then each thread will be responsible for 100 iterations. openmp was initially designed with for loop in mind.

OpenMP does not deal with while loop! this is because when while loop starts, we do not immediately know how many iterations there are.

OpenMP is not designed for recursion - have to translate recursion to for loops.

OpenMP can parallelize many serial programs with **relatively few annotations** that specify parallelism and independence

OpenMP is a small API that **hides cumbersome threading calls with simpler *directives***.
> built on top of C and fortran
> **directives** are anything that starts with a `#`
