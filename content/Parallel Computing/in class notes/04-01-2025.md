# OpenMP

An **API** for shared-memory parallel programming

> want a language that is close to the machine (which is never possible for an interpreted language like python), or java where we cannot control garbage collection, speed of JVM, etc.
> in the future, a language like Rust is coming up very quickly, so we may be using that. But for now, OpenMP is on top of C, C++, and Fortran

> OpenMP is for shared-memory, so we are talking about **threads** and not **processes**. We are taking one process, and parallelizing between many threads.

Designed for systems in which each **thread** can have access to all available memory

System is viewed as a collection of cores, all of which have access to the same main memory --> **shared memory architecture**

> can be viewed like your laptop: a collection of cores, all have access to the same main memory (e.g. 8 or 16 or 32 gb of ram).

![[shared memory system.png]]

## Pragmas

Most of OpenMP is based on Pragmas

> Most of the OpenMP code start with `#pragma omp`

`#pragma`

> any thing that starts with `#` is a comment to the preprocessor. Any code first reaches pre-processor before compiler. Preprocessor removes all the comments, extra spaces, removes and executes that start start with `#` (like `#include<std.h>`, etc. where it will grab all the code from `std.h` and copy it into your code, then remove this line from the code).

Special **preprocessor** instructions

Specified by the C standard for providing additional information to the compiler, beyond what is conveyed in the language itself

> `#` is specified by C standard, but `#pragma` is for OMP.

## Example: The Trapezoidal Rule
![[trapezoidal rule.png]]

Assignment of trapezoids to threads
![[trapezoidal rule thread assignment.png]]

![[thread problem aggregating sum trapezoidal rule.png]]

Suppose we have only two threads, each one executing in different core (Thread 0, thread 1). Suppose each core has its own L1 cache. And then we don't have any other caches, but we have the main memory (physical memory) that has `global_result`. Thread 0 will need to execute `global_result+=my_result`, and thread 1 executes the same thing. Both of them needs to get `global_result` first.

Each one will look in its cache, and it will not find `global_result` there --> cache miss. THey both go to global memory, bring `global_result` to each cache. So far so good.

Q: At that point, how many copies of `global_result` do we have?
A: 3 - one in each cache, and in the original one

Q: At that moment, where the coherence hardware kick in, or not?
A: no. coherence kicks in when update happens. 

When you want to add `global_result += my_result`, note that the hardware does not understand variables. This is why when you studied assembly in 201, you see that there is no variables only registers. variables are only created just for you. In order to add the two numbers, we need to get the two numbers to the registers inside the core.

At time 1, `global_result` that was initially 0 will go to a register that in thread 0's core. Thread 1 is still calculating `my_result`.
 `global_result` - the copy in the other cache, will go to a register in the other core. Now, two registers in first core have `global_result` and `my_result` (thread 0). In thread 1, `global_result=0` in one of the registers.

 At time 2, thread 0 adds `my_result` and `global_result`. at the same time, thread 1, whicih already has `global_result=0` in register, and thread 1's `my_result=2` moves from cache to register too. in thread 1, one register has `global_result=0`, and another register has `my_result=2`. I have also already added both `0+1=1` (in thread 0)

 **The coherence *will not kick in***, because we have not written anything into the cache

 At time 3: the result is still in the core. Now we need to write this `global_result=1` into the cache block. At the same time, thread 1 is still one step behind is doing `global_result += my_result`.

 At this step, the cache coherence **will*** kick in. It will tell the cache in thread number 1 to remove the block that contains `global_result` and become invalid there. And the `global_result` in that cache will become 1. In thread 1, the `global_result` in the cache will become invalid

 At time 4, thread 0 is done. therad 1 need to write `global_result`. But `global_result` is not there - so there is a cache miss. Depending on the design of processor, it will either get it from the other cache, or the other cache will write to the memory and it will grab it from the memory. But now, it will write the result, which is `2`, back to the cach eblock. 
So, `2` will be write to the cach eblock.

The end result of `global_result=2`, which is wrong.
***note: we covered this before lol***

This tells us that when we have several threads of the same process executing something liek this, where global result is shared among the threads, cache coherence **will not save us**. We need something in sfotware to ensure correctness.

This is why we invented so many things like locks and stuff. Coherence by itself will not save us.

What do we do in software to ensure correctness?

## Example 2

**problem**: count the number of times each ASCII character occurs on a page of txt
**Input**: ASCII text stored as an array of characters
**Output**: A histogram with 128 buckets - one for each ASCII character

> note: lowercase a ≠ uppercase A

Before we see what is coming, suppose yo uhave athis problem in front ofyou, one of the thing you have to start thinking about is what is the best way to parallelize this code. First, what are all the strategies that we can parallelize this code with among threads - think about anything

- Option 1: lets' cut the big string into equal pieces as much as we can, and get each piece into a thread
- Option 2: each thread will be responsible for one or more buckets of the 128 buckets.

Either we divide the text among the threads, or divide the buckets among the threads

Each one of the two strategies will give parallelism (some speed up). But, scalability is important. As the problem gets bigger - as string gets longer and longer - which one of the two strategies will work?

**A sign of a good parallel program is that when problem size gets bigger, you can (its better to) increase the number of threads, but not increase the number of work per thread.** 

Applying this to the two strategies:
1. cutting text into pieces. if text gets longer, we can cut into more pieces, and each thread will still work with the same length in each piece
2. with option 2, every thread will have to scan the whole string to fill out its assigne dbucket. we cannot increase the amount of thread, but the amount of work done by each string will increase. The second strategy is not scalable.

We will be dividing string into pieces, and each piece is assigned to a thread, and get one histogram at the end.

![[ascii histogram sequential version.png]]

`compute_histogram_st`: starts with `*page, page_size, *histogram`
- `*page` is a ASCII string
- `page_size` is the size of the string
very straightforward. if you run this sequential code, speed on quad core: 10.36 seconds

We need to aprallelize this
![[ascii histogram parallelism visualization.png]]

```C
void compute_histogram_st(char *page, int page_size, int *histogram) {
	# pragma omp parallel for
	for (int i = 0; i < page_size; i++) {
		char read_character = page[i];
		histogram[read_character]++;
	}
}
```

Note: after `# pragma omp parallel for`, the next line **must** be a for loop.

> if we have 12 iterations, and 3 threads, that means each thread will take 12/3=4 iterations. the first thread will take 0-3, second thread takes 4-7, and so on.

Remember that the sequential code took 10.36 seconds. Assume we are running this code in a 4-core machine, each one is only one way hyperthreading - so we only really have 1 logical core.

**oh... the code won't work! Look at line number 5**

look at line 5: `histogram[read_character]++;`
this is a critical section!

```C
void compute_histogram_st(char *page, int page_size, int *histogram) {
	#pragma omp parallel for
	for (int i = 0; i < page_size; i++) {
		char read_character = page[i];
		#pragma omp atomic // <--
		histogram[read_character]++;
	}
}
```

the `atomic` will ensure no two threads will be in the critical section line at the same time.
> `atomic` is a bit faster than `critical` (or whatever its called), and we will cover this later

The code **is correct**. but what about performance

We will get **114.89 seconds**, which is **10x slower than the single thread version!!!**

Reason: you are trying to execute the iteration in parallel, but inside the iteration it is reading a character and then updating the histogram. By putting `#pragma omp atomic`, you make updating histogram a critical section. Simply speaking, this code basically run sequentially, but then PLUS the overhead of creating threads and what not. It is correct, but its performance is pretty bad.

Can we adjust our algorithm so that we do not have as many critical section?

![[openmp ascii histogram code v2.png]]

Now, we have something called `local_histogram` - it is two dimensional. In the first dimension, it is the thread number, the second dimension the buckets.

In the paralle region (inside the bracket), every thread gets an id. Then, we enter for loop
(forget about `nowait` for now)

every thread will be reading part of the characters (from `page`)
`local_histogram[tid][read_character]++`: every thread, based on its own id, will increment its own histogram

Note that, we have `#pragma omp for` and NOT `#pragma omp parallel for`. This is because we have `parallel` outside! - all the threads would have already been created and alive by this line. the `parallel` is alive from line 3 to line 15.

The word `for` just says that - now, i have a for loop coming after, and i want from you to divide the loop among the iterations that are currently existent. `#pragma omp for` doesn't create theads, but just use the threads that are already created and divide the for loop among them.

`nowait`: usually, since every thread gets subset of iteration, it is possible one thread finish before and remember that all the threads are alive until line 15. usually, openmp puts an implicit barrier at the end of the for loops when we use `pragma omp for`, so no thread will be able ot process until all the threads reach line 10. But, if you put `nowait`, you are telling openmp runtime to remove the implicit barrier. Whenever a thread finishes, it will proceed. 

It will proceed to line 11, where every thread is using its id and all the characters to add to the global histogram. Take into account that since we remove implicit barrier and other stuff, each thread will execute the whole (second) for loop. more than one thread may add to the same `histogram` at the same time, so we add `pragma omp atomic`.

Q: is this code correct?
A: yes

Q: what about the performance? still quad core, sequential version took 10.36
A: it runs in **3.8 seconds**. But, speedup is *not* 4 yet! 10.36/4 --> we are supposed ot be getting a smaller second.

Note, here we are only measuring time of parallel part - we do not have any sequential part (we don't count them). Can we try to enhance this code a little bit?

![[openmp ascii histogram v2.1.png]]

the red line: its nothing to do with openmp. it just tells program - i want the beginning of the address of the 2d array to start at somewhere that is a multiple of 64. 
- This makes everything will be well aligned. it may or may not lead to high performance, but it won't hurt the performance.

OpenMP will not be able to proceed after line 10 (we have the implicit barrier after the for looop (no `nowait`))

Line 12: `#pragma omp single`
It means that the following structured code must be executed *only once* - we do not need to parallelize this. whichever thread reaches this first, does it and thats it.

This is sequentially, but we do NOT have the atomic anymore. However, we do not want any thread to add `local_histogram` to `histogram`, without knowing that all the other threads have finished working! thats why we need the implicit barrier

**speed is 4.42** - slower than the previous version!
