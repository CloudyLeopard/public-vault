## Is Any Application Suitable for GPU?

obviously, *no*. if all application is suitable for GPU, then a laptop should probably just contain GPUs.

GPUs are quite good for a subset of applications, and very bad for others. The type of chips that have this characteristics are called **accelerators**. 
- other examples: TPUs, FPGAs, Quantum Computing...

You will get the best performance from GPU if your application is:
- computation intensive
> no idle

- many **independent** computations
> GPUs get good performance from *vast* performance, which means independent.

- many **similar** computations
> not just independent, but similar. As we already hinted long time back at when we talked about Flynn taxonomy, these are SIMD - we can call them STMD (single thread multiple data) too. Run SAME subset of code on a large number of data

- problem size is big enough
> in order to execute on GPU, we have to bring data from system memory (connected to multicore) to GPU memory. This is a *very slow operation*. in order to get performance overall, you have to have very high performance on GPU overall to overcome the cost of this slow operation. this will never happen unless the problem size is big enough.

### Scheduling In NVIDIA GPUs

Q: If you make your decision of how many blocks to have, do you thikn you are restricted to the number of SMs on the GPU? (e.g. you must not have more than 8 blocks when yoeu xecute your code suppose GPU only has 8 SM) 
A: Of course not, because if i have 8 SMs, what if after i finish i want to send it to peopl eand people will be running your code with different SMs?
Also, just because i have 8SMs, doesnt mean i can only execute 8 blocks at the same time. for one, once an Sm finish a block, it will start putting a new block in there; or, if the Sm is big enough, it can do more than on eblock at a time.

Number of blocks is totally independent from the number of SMs. GPU will assign them. We decouple the nuber of blocks from the number of SMs.


Now, lets look at execution unit inside the SMs.
Q: how many threads per block? Can you have a block that has more threads than the number of SPs? Here is the trick: if you say no, we also have the portability problem - what if someone eslse takes your code and runs it on a computer that has different SPs/SM?
How do we decouple the number of threads per block from the number fo SPs/SM?
A: NVIDIA did something qutie smart here. they said that they will take numer of threads per block that programmer decides, and divide them into small groups of 32 threads each (ofc the last group will have less if its not divisible). This number is called **warp**.

If you decide to have 64 threads per bock, once a block is assigned to an SM, it will divide it to 2 warps. If you have 65 threads, it will have 3 warps: 32 + 32 + 1
NVIDIA ensures that no SMs have less than 32 SPs. And, all SMs in any GPUs will have number of SPs that are multiple of 32. 
Another thing: since these SPs are just execution unnits, they need a piece of hardawre that does fetch and decode and what not. NVIDIA decided that every 32 SPs have the same **frontend** (the hardware that does the fetch and decode). in this case, if you decide that if you have 64 threads per block, and you execute code on a GPU that has only 32 SPs per SM, once the block is assigned to the SMs it will be divided to two warps: warp 0 will have first 32 threads, warp 1 will have another 32 threads. The first warp will be asigned to first 32 SPs. once its done, the other 32 wil be assigned. sometimes, when a warp is assigned it cna be remove dbefore it is finished it does long latency oepration that needs to access memory or osmething, and other warp can proceed while the first warp is waiting for memory to come.

Given this description, we can think about how warp is executed. If every 32 SPs share the same fetch and ecode and issue, that means one instruction is fetched and decoded and sent to the SPs they'll execute, and then the secod istruction is fetched and decoded and sent to all of them. this means every 32 SPs will be executing in a lock step: all of them execute one isntruction, then the next, and so on.

What also does this mean? it means that the warp will take 32 SPs, regardless of how many threads is in that warp. So, if you have 33 threads per block, the hardware will divide it into 2 warps: warp 1 will have 32 threads, warp 2 will have 1 thread. Even in warp 2, 1 SP will work on on ethread, but the other 31 SP will be idle (cuz they are all conencted to the same hardware)
This tells us that when we make blocks, better make them **as close as possible to multiples of 32**.



**two level**, distribuetd to thread scheduler
- at the chip level (level 1): a global work distribution engine schedules thread blocks to various SMs
- At the SM level: each warp schedule distribuets warps of 32 threads to its executin unitss

This idea of warps decouples the number of threads per block from number of SPs per SMs. also allow NVIDIA to leave a lot fo space in SMs for execution unit: now i will have decode and fetch for all 32 SPs; do not need one for each SP.

Modern GPus can simultaneously execute **multiple kernels of the same application**
> if you are writing code in OpenMP, and you have several threads, each one of these threads can send a kernel to the GPU at the same time

Two **warps** from different blocks (or even different kernels) can be issued and executed simultaneously

MOre advanced GPUs can do more than that, but we will concetrate on the aboev only here

> warps are programmer transparent, which means we will not fid every API that will control warps. but knowing warps will affect our programming. its like cache - its programmer transparent, but knowing how it works w ecan write bout cache friendly code
> knowing abt warp, we know **we want number of threads per block to as close as possible to multiples of 32**.

another very important thing: suppose we have a conditional branch insider kernel (if else), and we have a warp. And all the SPs reach the instruction fo the If condition. it can happen that some of the thread will have the condition TRUE, some will have condition FALSE.

but, since now they have the same front end, they execute the same instruction, how will something liek this happen?

A: (inside the same warp): instructionsn that hav condition true will execute the true stuff, the other SPs will be idle. Once they are done, the SPs will start executing the else step.
This means, youe xecute the if and else *sequentially*, NOT IN PARALLEL. so you lost performance.

once of the trickiest thing in GPU is this problem, called **branch divergence** or **thread divergence**, because it makes you lose perofmrance. in kernel, whenever you have if-else, have to think will this cause branch divergence.

We will learn about how to recognize branch divergence, and how to deal with it.

## The Memory Hierarchy

Multicore: Level 1 with each core, Level 2 in each core, then Level 3 shared

Inside each SM:
- first level L1 cache
- shared memory (under your control)

GPUs are also equipped with an L2 cache, shared among all SMs.

No L3 cache, no branch predictio in front end, no coherence hardware on GPUs
> if two threads write to the same thing, chaos will happen and its totally up to you to deal with it. They are trying to remove any unnecessary hardware to make vast majority of chip space to be for execution units.

The last level is the **global memory** (memory of GPU card)
