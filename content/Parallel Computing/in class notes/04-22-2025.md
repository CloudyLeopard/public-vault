
# CUDA

Code that is GPU friendly has to satisfy four conditions (see lecture from last week)

Once you decide which part of the code will be out sources to the GPU, you put that in a function (called a kernel) and send that to the GPU. The full fledged function can have more than one kernel.

Using NVIDIA language, the multicore is called the **host**, and the GPU is called the **device**. 

Usually, you start with the main function starting with the CPU. then, some part of the code may be purely parallel but not GPU friendly. For example, lots of independent and computation intensive code, but code is not similar, so we may use OpenMP.

A kernel is like a function call with a few exception: its like a function call in the sense that it has a name and bracket that contains argument. But, between function name and arguments and what not it has triple < and > and variables in betwee
```
KernelA<<< nBlk, nTid >>>KernelB
```

Kernel
- launched by the host
- very simlar to a C function
- to be executed on device
- all threads will execute that same code in the kernel

> piece of code that will go to GPU. once it goes to the GPU, its no longer called a kernel; its called a "grid". Its like an executable being called a process once it starts executing.

Grid:
> bunch of blocks; every block will be assigned to an SM.
> depending on resources, every SM can hold one or more block at a time. Block will stay in that SM until its done executing the whole block.

> we said that a grid is a group of blocks. But we need to be able to designated a block. there must be some indexing. The indexing in CUDA allows you to organize the blocks in the grid in a 1D, 2D or 3D organization. 

- 1D, 2D or 3D organization fo a Grid
> we said before that 1D, 2D, 3D does not affect performance. it is used to make life easier fror you. suppose you are doing matrix multiplication, obvioulsy 2D; if you are doing vector additon, obviously 1D.

- gridDim.x, gridDim.y, gridDim.z are the size of the grid in number of blocks

> once you decide on 1D, 2D or 3D, you have 3 variables you can use inside yoru code without ANY initialization. gridDim.x = how many blocks in the x direction; gridDim.y = how many blocks in the y direction, etc. If you decide 1D, then gridDim.y=gridDim.z=1. If you want 2D, then gridDim.x and gridDim.y will have the numbers, and gridDim.z=1. In all cases, you can use these variables **right away without any initialization**.

Block:
> block conaints thread, so we must also be able to index them

- 1D, 2D or 3D organization of a block
- block is assigned to an SM
- blockDim.x, blockDim.y, blockDim.z are block dimensions counted as number of threads
> no initialization needed

- blockldx.x, blodldx.y, blockldx.z are indices of the blockw thin a grid
> in MPI, threads have ids; in OpenMP, threads have ranks. there, every block has an id. its called `blockIdx`. if yo udecide that the block in the grid is 1d, then `blockldx.y` and `blockldx.z` are always equal to 1.

Thread
- threadIdx.x, threadIdx.y, threadIdx.z are the index *within* a block.

### Decisions you have to make as a GPU programmer

1. which Part(s) of the program will be executed on the GPU?
> need to determine the kernels
2. how many total threads will you spawn
3. how many blocks? that is, how many threads per block?
4. What iwll be the geometry of the block (1D, 2D or 3D)?
5. what will be the geometry of the grid (1D, 2D or 3D)?

if you make this five decisions well (not just correctly), then you will get much higher performance then if yo urun it on the multicore.

### Blocks

Divide monolithic thread array into multiple blocks
- threads within a bock cooperate via **shared memory**
- threads in different blocks cannot cooperate except through global  meomry

> threads in different blocks cannot use shared memory, has to use global memory or memory of the GPU card.

### CUDA Device Memory Allocation
```C
cudaMalloc()
```
- allocates object (any data structure) in the device **global memory**
- requires two parameters
	- **address of a pointer** to the allocated object
	- **size** allocated object

```C
cudaFree()
```
- Free objects fromd evice Global Memory
- Parameter: pointer to freed object

For example, if you want to allocate an array of 64 x 64 floating point matrix:
```C
WIDTH = 64;
float* Md
int size = WIDTH * WIDTH * sizeof(float);

cudaMalloc((void**)&Md, size);
cudaFree(Md);
```

just to make `cudaMalloc` very generic, they typecast all the addresses with `void**`. after that, you can just use it like a real pointer.

So, simply, this is the size and then uc an use `Md` like a one dimensional array (`Md[...]`)

This will help us in part 1, which is allocating space.

### More on Specifying Dimensions

When yo ucall the kernel, there is the trainge thing rememebr
Now, we know programmer wants to use 2D block. how do we specify in <<< and >>> that we are using 2D?

```C
// Set up the execution configuration
// this is outside your code; this is also outside of CUDA Lol you can call it whatever
dim3 dimGrid(x, y, z);
dim3 dimBLock(x, y, z);

// launch the device computation threads!
MatrixMulKernel<<<dimGrid, dimBlock>>>(Md, Nd, Pd, Width);
```

Important:
- dimGrid and dimBlock are user efined
- gridDim and blockDim are built in predefined variable accessible in kernel functions

`dim3` is a type defined in `cuda.h` that is a small structure that contains three numbers, and its only used to tell CUDA runtime that we have 2D or 3D. (two numbers means 2D, but i assume then its `dim2`)
