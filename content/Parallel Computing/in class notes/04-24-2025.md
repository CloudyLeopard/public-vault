## Software <-> Hardware

From a programmer's perspective...
- Blocks: all the blocks will execut ethe exact same code of the kernel, but different data
> how do they get different data? unique id, `gridDimX`, `blockIdX`, `threadIdX`...
- Kernel: the part of the code that will be executed from the GPU (the function) 
- Threads
- Grid: once the kernel starts executing on GPU; it is a group of blocks

Hardware Implementation:
- SMs: every block is assigned to an SM. it will stay inside taht SM until SM finishes execution of that block.
- SPs (per SM)
- Warps
> any execution unit cannot exist by itself; it needs some hardware that fetches instruction and what not. NVIDIA decided that every 32 SPs inside an SM will share the same "frontend" (fetch code that issue, and instruction will go to the 32 SPs.)
> If SM contains 128SP, that means there are 128/32=4 frontends. Those groups of 32 threads executing on the SP sharing the same frontend are called **warps**

There are two advantages to warps:
1. Don't require frontend for every SP. leave more space in chip to leave more SP; more parallelism
2. decouples relation between the number of threads per block and number of SPs per SM.

## Some Restrictions

- all threads in a grid execute the same kernel code
- A grid is organized as a 1D, 2D or 3D array of blocks (gridDim.x, gridDim.y, and gridDim.z)
- each block is organized as 1D, 2D or 3D array of threads (blockDim.x, blockDim.y, blockDim.z)
- Once a kernel is launched, its dimensions cannot change
> dimension is the total number of threads, total number of blocks; that means how many blocks, how many threads per blocks, and how they are organized (1D, 2D, 3D). these are fixed and cannot change dynamicaly when lanched
- All blocks in a grid have the same dimension
- the total size of a block, in terms of number of threads, has an upper bound
- once assigned to an SM, the block must execute in its entirety by the SM

## Synchronization

```C
__syncthreads() # two _ 
```
You cannot synchronize across all the threads in the grid. the threads in one block can only synchronize together, but cannot synchronize across blocks.

- called by a kernel function
- the thread that makes the call wil be held at the calling location until every thread in the block reaches the location

Q: why nvidia engineer there are no global barriers? (cannot synchronization across the grid)

suppose hypothetically NVIDAI allowed for global barriers, that means all the threads running teh SM will call that barrier? - all the threads in all the blocks in that SM will be stuck at the barrier, waiting for the rest of the threads in the other four blocks in the ready queue to reach the barrier. but those blocks in the ready queue are not executing - they do not have any SMs available right now. They never run `__syncthreads`, and we hvae a deadlock

**beware** of if-then-else
- if you put `__syncthreads` inside the else, and another `__syncthreads` inside the if, these are not considered the same `__syncthreads`
- so if part of kernels reach the one in if, the others reach else, they still nothing. **undefined behavior

**threads in different blocks cannot synchronize!**

## Scheduling of Blocks

To be assigned to an SM, a block needs to have all its resources (registers, shared memory, number of threads, ...) assigned beforehand.

> sometimes a block gets broken up into warps or something (cuz different size)
> so when i remove something and put in something, it needs to happen simultaneously. i can just swap in and out.

- CUDA runtime automaticalyr educes number of blocks assigned to each SM until **resource usage** is under limit
> SM has enoughr esources to have 2.5 blocks, so we assigned 2 blocks
> if it only has resources for 1 block, ony one block will be assigned.
> if it only has resources for 1 block + 99%, then still only 1 block wil be assigned.

Runtime system:
- maintains a list of blocks that need to execute
- assigns new blocks to Sm as they compute previously assigned blocks.
> when SM finishes executing a block, they will get another block.

> this slide makes many swe wonder "what happened" lol
> sometimes, people who do not know how hardware works, and it works. but then they add one more variable - no drastic change. once they did this, they realize - the performance dropped by 40%! Just by adding one variable!
> When we add this one variable, `int x`, that means every thread will add an `intx`. This is because `x` is a scalar. if we make `int x, y`, every thread will have its own `int x, y`. the number of registers increased! it needs registers for `x`, and `y` to accomodate for each block. by adding one variable, its made the whole thing slower.

## Warps

Once a block is assigned to an SM, it is divided into units called warps
- thread IDs within a warp are consecutive and increasing
- warp 0 starts with Thread ID 0 till thread ID 31, and so on.

> NVIDIA has imposed this for all this time: the number of SPs inside SM is never less than 32; the number of SPs per SM is ALWAYS multiple of 32.

> lets say you have an Old GPU, wher enumber of SPs/SM are 32 SPs. sAssume that the programmer decides to have the block of 128 threads. How many warps are formed?
> 128/32 = 4

> lets keep the number of SPs/SM 32.
> if programmer decides to have a block of 50 threads. how many warps?
> 2! one warp has first 32, the other warp has 18. but remember, when a warp is assigned threads, you take the whole 32 SPs of the warp. yo ucannt only take half a warp. so, you are underutilizing the SPs.
> cuz ull have 14 warps in teh SP doing nothing.

As a programmer, try to make the number of threads per bock as close to 32 as possible. Li,e i fyou have 33, then you will still have 2 warps, but one warp will only use 1 Sp and 31 idle SPs.

suppose you have 4 warps. which warp starts executing first? which warp finishes last? these are undeterministic

Warp size is implementation specific
- but so far, all NVIDIA GPUs have warp of 32 threads

Warp is unit of thread scheduling in SMs.

- Partitioning is always the same (first 32 threads goes to firs twarp, etc.)
- We cannot determine which warp finishes first

- Each warp is executed in a SIMD fashion (i.e. all threads within a warp must execute the same instruction at any given time)
	- Problem: **branch divergence**
> Small rpoblem: each warp is executed as SIMD. The SPs are execution units - they have teh same frontend. So, we fetch one instruction, decode it, then send it to 32 SPs to execute, as if the 32 SPs are executing the instruction in a lock step - as if there is a barrier after every section. they will do all together every seciton.  If a warp has like 5 threads, that means 5 SPs will eb executing in a lock step, the other 27 SPs will be doing nothing.

> branch divergence; we said all instruction is executing in a lock step. but what if an instuction is an if statement? that means some of the thread in warp will reach condition to be true, other thread be false. but since if all the SPs must execute the same instruction, what will happen?

### Branch Divergence in Warps

occurs when threads inside warps branches to different execution

(suppose we have 32 arrows in the diagram below)
Some of the threads will go into PathA, the other threads are doing nothing and just consuming power. When these are done, the other threads will start executing and teh first few threads will wait and start consuming power.

![[GPU branch divergence.png]]

We executed if and else "sequentially" - **50% performance loss.**

from a progamming pov, whenever we are writting in if-else inside kernel, start thinking is there a situation where i will have branch divergence. 

based on this hardware description:
1. try to make number of threads per block as lock as possible to multiples of 32
2. be aware of number of scalar variables, ebcause they may affect your kernel
3. be very careful whenever u put an if condition.

## Latency Tolerance

When an instruction executed by the threads in a warp must wait for the result of aprevioulsy initiated long-latency operation, the warp is not selected for execution -> **latency hiding**

> suppose whe you block is assigend to SM, and divided into 4 warps. the number of Sp in SM can execute two warps at the same time. And then, suppose one of the warp needs to access global memory, which is slow
> instead of keeping the warp there on teh SPs waiting for global memory (whic his a long latency oepration), the hardware of SM will remove the warp and will put one of the two warps that are waiting in to be executed.

> another thing, if you have a brrier - a synchronization point - two warps will be there adn execute until they reach the barrier. once they reach the barrier, the cuda runtime will remove them and put in the two other warps. Now, all 4 warps have reached barrier ad they can conintue

> having long latency operation we can remove warp and put in a nother warp and hide the latency

> usually, when u have to wait, u can 1. speed it up (i.e. use more advanced technology to speed up memory access) or 2. hide it (while im watiing for memory, i will not just be idle; i will execute something else useful).

In some books, latency hiding is also called **latency tolerance**. I am not just waiting, but doing something useful.

By moving a warp and putting another warp in, takes almost 0 cycles. this is because **we do not have to do any context switching**. 

Shceduling does not introduce idle time -> **zero-overhead thread scheduling**
> sm gets all the resource it needs at th ebeginning

Scheduling is used for tolerating long-latency operations.
